**Goal:** Add API key authentication for processing services, pass client_info on every request for instance tracking, and refactor /tasks and /result endpoints with proper DRF serializers.

## Architecture

### Authentication Model

ProcessingService has an `api_key` field (prefixed `ant_ps_`, generated by `secrets.token_urlsafe(36)`).

A custom DRF auth backend (`ProcessingServiceAPIKeyAuthentication`) reads `Authorization: Bearer ant_ps_...`, looks up the PS, and returns `(AnonymousUser(), processing_service)`.

The PS identity lives on **`request.auth`**, not `request.user`. This follows the `djangorestframework-api-key` library pattern. See [Research: API Key Auth Patterns](#research-api-key-auth-patterns-in-drf) below.

### Permission Model

A dedicated `HasProcessingServiceKey` permission class (in `ami/base/permissions.py`) checks `isinstance(request.auth, ProcessingService)` and verifies the PS is linked to the relevant project via its `projects` M2M.

Endpoints used by both users and services compose permissions:
```python
permission_classes = [ObjectPermission | HasProcessingServiceKey]
```

DRF 3.14's `|` operator provides short-circuit evaluation — if either side passes, the request is allowed.

### Client Info

Every PS request can include `client_info` (hostname, software, version, platform, pod_name, extra). The server merges this with observed IP and User-Agent. Stored on `ProcessingService.last_seen_client_info` as JSON.

### Breaking Changes

These changes require updates to the ADC worker (`ami-data-companion`):

1. **`/tasks` is now POST** (was GET). Body: `{"batch": N, "client_info": {...}}`
2. **`/result` accepts wrapped format**: `{"client_info": {...}, "results": [...]}` (bare list still accepted for backward compat)
3. **`/pipelines/` registration**: `processing_service_name` optional when using API key auth (PS identified by key)
4. **Authentication**: Workers should send `Authorization: Bearer ant_ps_...` instead of `Authorization: Token ...`

## Research: API Key Auth Patterns in DRF

This section documents research done before Task 2.5 to inform the permission model design.

### The Core Question

DRF's `authenticate()` returns `(user, auth)`. For service-to-service API key auth where there's no real Django User, what should `user` be?

### How Other Libraries Handle It

**`djangorestframework-api-key` (most popular)**
- Implements API keys as a **permission class** (`HasAPIKey`), NOT an auth backend
- Does NOT set `request.user` — it remains `AnonymousUser`
- `IsAuthenticated` fails for API-key-only requests (by design)
- Composition: `permission_classes = [HasAPIKey | IsAuthenticated]`
- Explicit guidance: "you should NOT use this package to identify individual users"

**`rest_framework.authentication.TokenAuthentication` (built-in)**
- `request.user` = the Django User the token belongs to
- `request.auth` = the Token model instance
- User-centric; no mechanism for service-identity tokens

**`django-oauth-toolkit` (client credentials)**
- OAuth2 client credentials for machine-to-machine
- Apps without associated user → `request.user` = `AnonymousUser` or `None`
- Known pain point (issues #38, #1155)
- Overkill for simple internal service auth

### The AnonymousUser Approach

A documented pattern: return `(AnonymousUser(), service_token)` from `authenticate()`. The service identity lives entirely on `request.auth`.

**Gotchas:**
- `IsAuthenticated` returns False (AnonymousUser.is_authenticated = False)
- Code that assumes `request.user.pk` is a real user will break
- Django admin, audit logs, middleware that touches `request.user` as a full User

**Mitigation:** Write a dedicated permission class that checks `request.auth`, compose with `|`:
```python
permission_classes = [IsAuthenticated | IsProcessingService]
```

### Our Approach

We combined both patterns:

1. **Auth backend** sets `request.auth = ProcessingService` (the real identity)
2. **`HasProcessingServiceKey` permission** checks `isinstance(request.auth, ProcessingService)` + project access
3. **Per-action composition** on PS-facing endpoints: `[ObjectPermission | HasProcessingServiceKey]`

This avoids: dummy User records, `has_perm()` crashes with django-guardian, header conflicts with djoser's Token scheme.

### Header Convention

- Djoser users: `Authorization: Token <djoser_token>`
- Processing services: `Authorization: Bearer ant_ps_<key>`
- The `ant_ps_` prefix ensures early bail-out — if the token doesn't start with the prefix, the auth backend returns `None` and DRF falls through to `TokenAuthentication`
